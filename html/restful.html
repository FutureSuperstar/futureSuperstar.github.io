我对restful的理解

restful是最近比较火的一个概念，restful架构有着结构清晰、符合标准、易于理解、扩展方便等诸多的优点，所以正得到越来越多网站的采用。但是，到底什么是RESTful架构，并不是一个容易说清楚的问题。

REST这个词，是Roy Thomas Fielding在他2000年的博士论文中提出的。 REST是Representational State Transfer的缩写，翻译成中文就是具象状态传输。也就是传输具象或者传输其状态。

REST 定义了一组体系架构原则，您可以根据这些，包括使用不同语言编写的客户端如何通过 HTTP 处理和传输资源状态。 在事实上，REST 对 Web的影响非常大，由于其使用相当方便，已经普遍地取代了基于 SOAP 和 WSDL 的接口设计。

个人理解：
（一） REST只是一种规范或者说是风格，不是一种标准。
（二） REST强调的是资源，包括资源的获取，创建，修改，删除等。
（三） REST是建立在http的基础上。

在http中的方法有：

序号 方法 描述
1 GET 请求指定的页面信息，并返回实体主体。
2 HEAD 类似于get请求，只不过返回的响应中没有具体的内容，用于获取报头
3 POST 向指定资源提交数据进行处理请求（例如提交表单或者上传文件）。数据被包含在请求体中。POST请求可能会导致新的资源的建立和/或已有资源的修改。
4 PUT 从客户端向服务器传送的数据取代指定的文档的内容。
5 DELETE 请求服务器删除指定的页面。
6 CONNECT HTTP/1.1协议中预留给能够将连接改为管道方式的代理服务器。
7 OPTIONS 允许客户端查看服务器的性能。
8 TRACE 回显服务器收到的请求，主要用于测试或诊断。

当然了在表单中就只有get和post了。在rest中常用的方法有四个get、post、delete、put。这几个方法中有既安全也是幂等的，也有不都不具备的。

方法 安全性 幂等性
GET √ √
POST × ×
PUT × √
DELETE × √

怕有小伙伴不明白这里说的安全性和幂等性，所以再解释一下：
安全性 ：不会改变资源状态，可以理解为只读的；
幂等性 ：执行1次和执行N次，对资源状态改变的效果是等价的。

综合上面的解释，我们总结一下什么是RESTful架构：
　　（1）每一个URL代表一种资源；
　　（2）客户端和服务器之间，传递这种资源的某种表现；
　　（3）客户端通过上面的四个HTTP动词，对服务器端资源进行操作，实现"具象状态传输"。

最后说一下restful的规范。总结起来有以下几点；

在URL中体现资源，而非动作
就是说在url中只有名词，动词用http的四个方法来体现。
版本号
关于这点目前有两种做法，一是把版本号放到url中体现，另一种是方法是放到header中。
返回值
建议返回json的数据格式，便于跨平台，且数据量更小，另外javascript也可以解析。一般json格式类似于：
{ 'code' : 0, 'data' : { 'userId' : 1, 'userName' : 'xiaomin' }, 'serverTime': 14939939 }
权限控制
目前建议是给客户端一个token，并且放在header中识别。
